# minecraft-ktor

Architectury mod **compile service**. Accepts Kotlin source files via HTTP, injects them into a pre-configured [Architectury](https://architectury.dev/) project template, runs a Gradle build, and returns the compiled `.jar` artifacts for both **Fabric** and **NeoForge** loaders.

This service is an internal microservice — it has no auth and is not exposed to the public internet. It is consumed by `minecraft-api` over the cluster network.

---

## Architecture

```
POST /compile
      │
      ▼
┌─────────────────────────────────────────────────────┐
│  CompileService                                     │
│                                                     │
│  1. Copy /opt/mc-template → /tmp/mc-builds/{uuid}/  │
│  2. Overwrite gradle.properties (modId, version)    │
│  3. Generate fabric.mod.json + neoforge.mods.toml   │
│  4. Write LLM-provided Kotlin sources into common/  │
│  5. Generate loader entrypoint stubs (fabric/neo)   │
│  6. Run ./gradlew :fabric:remapJar :neoforge:remapJar│
│  7. Read output jars → Base64-encode → return       │
│  8. Clean up temp directory                         │
└─────────────────────────────────────────────────────┘
```

### Architectury template (`template/`)

The template is a multi-module Gradle project baked into the Docker image at `/opt/mc-template`. It targets:

| Property | Value |
|---|---|
| Minecraft | 1.21.4 |
| Loaders | Fabric + NeoForge |
| Kotlin | 2.1.0 |
| Architectury API | 13.0.6 |
| Gradle | 8.11.1 |
| Java | 21 |

Module layout:

```
template/
├── common/          # Cross-platform mod logic (LLM sources go here)
├── fabric/          # Fabric loader entrypoint (generated per request)
└── neoforge/        # NeoForge loader entrypoint (generated per request)
```

The LLM only generates `common/src/main/kotlin/` files. The fabric/neoforge entrypoints are boilerplate stubs generated by the service from the `mainClass` field.

### Docker image cache pre-warming

The Dockerfile has a dedicated `cache-warmer` stage that runs a full Gradle build on the placeholder template. This downloads and caches:

- Minecraft 1.21.4 game jars + mappings
- Architectury API, Fabric API, NeoForge, KotlinForForge
- All Gradle plugin dependencies

The pre-warmed cache is copied into the runtime image at `/opt/gradle-home`. Without this, the first build would take 10+ minutes to download ~1–2 GB of dependencies.

## API

### `GET /health`

Returns `200 OK` when the service is up.

```json
{ "status": "ok" }
```

---

### `POST /compile`

Compile Kotlin sources into Fabric and NeoForge `.jar` files.

**Request body:**

```json
{
  "modId": "fire_sword_mod",
  "modName": "Fire Sword Mod",
  "version": "1.0.0",
  "description": "A sword that sets enemies on fire",
  "mainClass": "FireSwordMod",
  "sources": [
    {
      "path": "common/src/main/kotlin/com/iguzman/mod/FireSwordMod.kt",
      "content": "package com.iguzman.mod\n\nobject FireSwordMod {\n    const val MOD_ID = \"fire_sword_mod\"\n    fun init() { /* ... */ }\n}"
    }
  ]
}
```

| Field | Type | Required | Notes |
|---|---|---|---|
| `modId` | string | yes | Lowercase, `[a-z0-9_]`, max 64 chars |
| `modName` | string | yes | Human-readable display name |
| `version` | string | no | Defaults to `"1.0.0"` |
| `description` | string | no | Short description |
| `mainClass` | string | yes | PascalCase name of the common entry-point object |
| `sources` | array | yes | One or more `{path, content}` objects |

**Source file contract** — the primary file must define:

```kotlin
object {mainClass} {
    const val MOD_ID = "{modId}"
    fun init() { /* mod logic */ }
}
```

**Success response `200 OK`:**

```json
{
  "success": true,
  "fabricJarBase64": "<base64>",
  "neoforgeJarBase64": "<base64>",
  "buildLog": "...(last 4000 chars of Gradle output)..."
}
```

**Failure response `422 Unprocessable Entity`:**

```json
{
  "success": false,
  "buildLog": "...(last 8000 chars of Gradle output)...",
  "error": "Gradle build failed (exit 1). Check buildLog for details."
}
```

## Environment variables

| Variable | Default | Description |
|---|---|---|
| `PORT` | `8080` | HTTP port |
| `TEMPLATE_DIR` | `/opt/mc-template` | Path to the Architectury template |
| `GRADLE_USER_HOME` | `/opt/gradle-home` | Pre-warmed Gradle/Minecraft dependency cache |
| `BUILD_WORKSPACE_DIR` | `/tmp/mc-builds` | Temp dir for per-request workspaces |

Copy `env.example` to `.env` to configure locally.

## Development

### Prerequisites

- JDK 21+
- Gradle 8.11+ (or use the wrapper: `./gradlew`)

### Run locally

```bash
# Copy env
cp env.example .env

# Start the Ktor server (hot reload enabled in dev mode)
./gradlew run -t
```

The server starts on `http://localhost:8080`.

> **Note:** On a local run without the pre-warmed Docker cache, the first `/compile` call will trigger a full Minecraft dependency download (~1–2 GB). Subsequent calls reuse `~/.gradle`.

### Build distribution

```bash
./gradlew installDist
./build/install/minecraft-ktor/bin/minecraft-ktor
```

### Test a compile request

```bash
curl -s -X POST http://localhost:8080/compile \
  -H "Content-Type: application/json" \
  -d '{
    "modId": "hello_mod",
    "modName": "Hello Mod",
    "mainClass": "HelloMod",
    "sources": [{
      "path": "common/src/main/kotlin/com/iguzman/mod/HelloMod.kt",
      "content": "package com.iguzman.mod\n\nobject HelloMod {\n    const val MOD_ID = \"hello_mod\"\n    fun init() {}\n}"
    }]
  }' | jq '.success'
```

## Docker

```bash
# Build (includes pre-warming — takes ~20 min on first build)
docker build -t christopherguzman/minecraft-ktor:latest .

# Tag and push
docker tag christopherguzman/minecraft-ktor:latest christopherguzman/minecraft-ktor:latest
docker push christopherguzman/minecraft-ktor:latest

# Run
docker run -p 8080:8080 christopherguzman/minecraft-ktor:latest
```

> The image is large (~2–3 GB) due to the Minecraft dependency cache baked in. This is intentional — it makes runtime builds fast.

## Deployment (Kubernetes / Helm)

```bash
helm upgrade --install minecraft-ktor ./helm \
  --namespace minecraft \
  --create-namespace
```

The service is deployed as a `ClusterIP` (internal only). The `minecraft-api` service reaches it at:

```
http://minecraft-ktor.minecraft.svc.cluster.local/compile
```

Resources are sized for compilation workloads (`2 Gi` RAM request, `4 Gi` limit). The build workspace uses an `emptyDir` volume for fast local pod storage.
