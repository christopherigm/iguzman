# minecraft-api

Django REST API backend for the Minecraft mod generation platform. Handles authentication, user management, and orchestrates the full **prompt → LLM code generation → compilation → artifact delivery** pipeline for Minecraft mods.

---

## Architecture

```
Client
  │
  │  POST /api/mods/  {"prompt": "A sword that lights enemies on fire"}
  ▼
┌─────────────────────────────────────────────────────────────────┐
│  minecraft-api  (Django + DRF)                                  │
│                                                                 │
│  1. Create Mod record (status: pending)                         │
│  2. Return 202 Accepted immediately                             │
│  3. Background thread:                                          │
│     a. Call Groq API → generate Kotlin sources (status: generating) │
│     b. Call minecraft-ktor /compile → get .jar files (status: compiling) │
│     c. Save .jar files to media/ (status: ready)                │
└───────────────────────┬─────────────────────────────────────────┘
                        │
          ┌─────────────┴─────────────┐
          ▼                           ▼
  ┌───────────────┐         ┌──────────────────┐
  │  Groq API     │         │  minecraft-ktor   │
  │  llama-3.3-   │         │  Ktor compile     │
  │  70b-versatile│         │  service          │
  └───────────────┘         └──────────────────┘

Client polls GET /api/mods/{id}/ until status = "ready" | "failed"
```

### Services

| Service | File | Responsibility |
|---|---|---|
| `groq_service` | `mods/services/groq_service.py` | Calls Groq API with a structured system prompt; returns parsed JSON with `modId`, `modName`, `mainClass`, `sources` |
| `ktor_service` | `mods/services/ktor_service.py` | HTTP client for the `minecraft-ktor` compile service; sends sources, receives base64-encoded jars |

### LLM prompt contract

The Groq system prompt instructs the model to return a JSON object with this shape:

```json
{
  "modId": "snake_case_id",
  "modName": "Human Readable Name",
  "description": "One sentence description",
  "mainClass": "PascalCaseClassName",
  "sources": [
    {
      "path": "common/src/main/kotlin/com/iguzman/mod/PascalCaseClassName.kt",
      "content": "..."
    }
  ]
}
```

The model only generates `common/` sources. Loader-specific entrypoints (Fabric/NeoForge) are boilerplate generated by `minecraft-ktor`.

## Data model

### `Mod`

| Field | Type | Description |
|---|---|---|
| `user` | FK → User | Owner |
| `prompt` | TextField | Original user prompt |
| `llm_model` | CharField | Groq model used (default: `llama-3.3-70b-versatile`) |
| `mod_id` | SlugField | Generated snake_case identifier |
| `mod_name` | CharField | Generated display name |
| `description` | TextField | Generated description |
| `main_class` | CharField | Generated PascalCase entry-point class name |
| `generated_sources` | JSONField | Raw sources returned by the LLM |
| `status` | CharField | `pending` → `generating` → `compiling` → `ready` / `failed` |
| `build_log` | TextField | Last ~4 KB of Gradle output |
| `error` | TextField | Error message if `status = failed` |
| `fabric_jar` | FileField | Compiled Fabric `.jar` (served via nginx from `media/`) |
| `neoforge_jar` | FileField | Compiled NeoForge `.jar` |

## API endpoints

All endpoints require **JWT Bearer authentication** (`Authorization: Bearer <token>`). Use `/api/auth/` to obtain tokens.

### Authentication

| Method | Endpoint | Description |
|---|---|---|
| `POST` | `/api/auth/register/` | Register a new user |
| `POST` | `/api/auth/login/` | Obtain access + refresh tokens |
| `POST` | `/api/auth/token/refresh/` | Refresh access token |
| `POST` | `/api/auth/verify-email/` | Verify email address |

### Mods

#### `GET /api/mods/`

List all mods belonging to the authenticated user.

**Response `200 OK`:**

```json
[
  {
    "id": 1,
    "prompt": "A sword that sets enemies on fire",
    "llm_model": "llama-3.3-70b-versatile",
    "mod_id": "fire_sword_mod",
    "mod_name": "Fire Sword Mod",
    "description": "Adds a sword that ignites enemies on hit.",
    "main_class": "FireSwordMod",
    "status": "ready",
    "build_log": "...",
    "error": "",
    "fabric_jar_url": "https://api.minecraft.iguzman.com.mx/media/mods/fire_sword_mod/fire_sword_mod-fabric-1.0.0.jar",
    "neoforge_jar_url": "https://api.minecraft.iguzman.com.mx/media/mods/fire_sword_mod/fire_sword_mod-neoforge-1.0.0.jar",
    "created": "2026-04-12T10:00:00Z",
    "modified": "2026-04-12T10:02:15Z"
  }
]
```

#### `POST /api/mods/`

Submit a prompt to generate and compile a new mod. Returns `202 Accepted` immediately — the pipeline runs in a background thread.

**Request body:**

```json
{
  "prompt": "A sword that sets enemies on fire",
  "llm_model": "llama-3.3-70b-versatile"
}
```

| Field | Required | Notes |
|---|---|---|
| `prompt` | yes | 10–2000 chars, plain English description of the mod |
| `llm_model` | no | Any Groq-compatible model ID. Defaults to `llama-3.3-70b-versatile` |

**Response `202 Accepted`:**

```json
{
  "id": 2,
  "prompt": "A sword that sets enemies on fire",
  "llm_model": "llama-3.3-70b-versatile",
  "mod_id": "",
  "mod_name": "",
  "status": "pending",
  "fabric_jar_url": null,
  "neoforge_jar_url": null,
  "created": "2026-04-12T10:05:00Z",
  "modified": "2026-04-12T10:05:00Z"
}
```

Poll `GET /api/mods/{id}/` until `status` is `"ready"` or `"failed"`.

---

#### `GET /api/mods/{id}/`

Return the current status and download URLs for a specific mod.

**Status lifecycle:**

```
pending → generating → compiling → ready
                                 ↘ failed
```

**Response `200 OK`** (when ready):

```json
{
  "id": 2,
  "status": "ready",
  "fabric_jar_url": "https://api.minecraft.iguzman.com.mx/media/mods/fire_sword_mod/fire_sword_mod-fabric-1.0.0.jar",
  "neoforge_jar_url": "https://api.minecraft.iguzman.com.mx/media/mods/fire_sword_mod/fire_sword_mod-neoforge-1.0.0.jar",
  ...
}
```

**Response `200 OK`** (when failed):

```json
{
  "id": 2,
  "status": "failed",
  "error": "Gradle build failed (exit 1). Check buildLog for details.",
  "build_log": "...",
  ...
}
```

## Development

### Prerequisites

- Python 3.12+
- pip

### Setup

```bash
cd apps/minecraft-api

# Create and activate virtual environment
python3 -m venv venv
source venv/bin/activate   # Windows: venv\Scripts\activate

# Install dependencies
pip install -r requirements.txt

# Copy and configure environment
cp env.example .env
# Edit .env — set GROQ_API_KEY and KTOR_SERVICE_URL at minimum

# Run migrations
python manage.py migrate

# Create a superuser (for admin access)
python manage.py createsuperuser

# Start the development server
python manage.py runserver
```

API is available at `http://localhost:8000`.
Django admin at `http://localhost:8000/admin/`.

### Running the full pipeline locally

You need both services running:

```bash
# Terminal 1 — minecraft-ktor (compile service)
cd apps/minecraft-ktor
./gradlew run

# Terminal 2 — minecraft-api (Django)
cd apps/minecraft-api
source venv/bin/activate
python manage.py runserver
```

Then submit a mod:

```bash
# 1. Register + login
curl -s -X POST http://localhost:8000/api/auth/register/ \
  -H "Content-Type: application/json" \
  -d '{"email": "test@example.com", "password": "testpass123"}'

TOKEN=$(curl -s -X POST http://localhost:8000/api/auth/login/ \
  -H "Content-Type: application/json" \
  -d '{"email": "test@example.com", "password": "testpass123"}' \
  | jq -r '.access')

# 2. Submit prompt
MOD_ID=$(curl -s -X POST http://localhost:8000/api/mods/ \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"prompt": "Add a glowing diamond sword that sets enemies on fire"}' \
  | jq '.id')

# 3. Poll until ready (typically 30–90 s after ktor warms up)
curl -s http://localhost:8000/api/mods/$MOD_ID/ \
  -H "Authorization: Bearer $TOKEN" | jq '{status, fabric_jar_url, error}'
```

## Deployment (Kubernetes / Helm)

### 1. Create `minecraft-api-secrets`

The Helm chart reads sensitive values from a Kubernetes Secret named `minecraft-api-secrets` in the `minecraft` namespace. Create it before deploying:

```bash
kubectl create secret generic minecraft-api-secrets \
  --namespace minecraft \
  --from-literal=secret-key='your-django-secret-key' \
  --from-literal=db-password='your-postgres-password' \
  --from-literal=groq-api-key='your-groq-api-key' \
  --from-literal=email-host-user='you@example.com' \
  --from-literal=email-host-password='your-smtp-password'
```

| Secret key | Env var injected | Description |
|---|---|---|
| `secret-key` | `SECRET_KEY` | Django secret key — generate with `python -c "from django.core.management.utils import get_random_secret_key; print(get_random_secret_key())"` |
| `db-password` | `DB_PASSWORD` | PostgreSQL password for the `postgres` user |
| `groq-api-key` | `GROQ_API_KEY` | Groq API key used for LLM code generation |
| `email-host-user` | `EMAIL_HOST_USER` | SMTP username (e.g. `you@example.com`) |
| `email-host-password` | `EMAIL_HOST_PASSWORD` | SMTP password for the email host |

> **Optional:** If your Redis instance requires a password, uncomment the `redis-password` entry in `helm/values.yaml` and add it to the secret:
> ```bash
> kubectl create secret generic minecraft-api-secrets \
>   --namespace minecraft \
>   ... \
>   --from-literal=redis-password='your-redis-password'
> ```

### 2. Deploy with Helm

```bash
helm upgrade --install minecraft-api ./helm \
  --namespace minecraft \
  --create-namespace
```

The `KTOR_SERVICE_URL` is pre-configured in `helm/values.yaml` to reach `minecraft-ktor` over the cluster network:

```
http://minecraft-ktor.minecraft.svc.cluster.local
```

All three services (`minecraft`, `minecraft-api`, `minecraft-ktor`) are deployed into the same `minecraft` namespace so they can reach each other via short cluster DNS names.
